## 記録の役割分担（what は git・why は開発記録）

- **コミットメッセージ**に「その版で何を変えたか（what）」を具体的に書く（版アップごとに必ずコミット）。
- **`開発記録*.txt`** は「なぜ・経緯・バグの原因と対策・既知の注意（why / how）」に絞って簡潔に書く。
  単純な変更点の羅列はコミットメッセージに委ね、二重記述を避ける。
- **`開発ロードマップ.txt`** は今後の計画を保つ。

## 運用（Git ワークフロー）

**最重要: 直接 `main` へ push しない。バージョンごとに PR → マージで履歴を残す。**
（バージョン単位のマージ履歴を残すことが、このリポジトリで GitHub を使う最大の目的。ユーザー指示 2026-06-29）

- **1 セッション ＝ 1 バージョン ＝ 1 PR。** そのセッションでバグ修正以外の変更を加えたら
  バージョンを **1つだけ** 上げ、セッション内の追加変更もすべて同じバージョン（＝同じ feature
  ブランチ）にまとめる。PR は1セッションにつき **1回だけ**。
- **PR はユーザーが明示的に指示するまで作らない。** 勝手に `gh pr create` しないこと。
  指示が来たら、その時点までの変更を1つの PR にまとめて出す。
- バグ修正のみのセッション（版番据置の「追補」）は **バージョンを上げない**。最新版ファイルを
  直接更新して追補として記録する（新しいファイルを作らない）。これも feature ブランチで作業し、
  指示時に PR にする。
- 流れ: ① feature ブランチを切る（例 `feature/<内容>` または `feature/<内容>-vX.YZ`）→
  ② 版アップ＋変更を `git commit`（what を明記・複数 commit 可）→ ③ **ユーザー指示で** PR 作成
  → ④ マージで1バージョン確定。push は feature ブランチに対してのみ行う。
- **版を上げたら**、この「現行ステータス」欄と `開発ロードマップ.txt` を更新する
  （`開発記録*.txt` は why／経緯がある場合のみ追記）。
